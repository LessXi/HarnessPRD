# 提示词套件模板

> 版本 1.0 · 2026-06-15
>
> 目标读者：AI 编程工具 + 人类开发者
> 文档目标：基于 PRD 和接口文档生成结构化提示词，让 AI 编程工具能直接根据提示词生成代码

├──────┬──────────┬──────────┬──────────────────────┤
│ 2.   │ 3.       │ 4.       │ 5.                   │
│ 功能 │ 接口     │ 前端     │ 代码审查              │
│ 开发 │ 实现     │ 开发     │ 指导                  │
│ 指导 │ 指导     │ 指导     │ （给审查者看）         │
│(后   │(后      │(后      │                      │
│ 端)  │ 端)     │ 端)     │                      │
└──────┴──────────┴──────────┴──────────────────────┘
     ↑            ↑            ↑
     └──── 数据来源：PRD + 接口文档 ────┘
```

## 1. 项目上下文

**用途**：给 AI 编程工具建立对项目的全局认知。任何开发任务开始前，AI 应优先加载这份提示词。不涉及具体功能实现，只描述"这个项目是什么"。

**数据来源**：PRD 第 2 章（产品概述）、第 3 章（MVP 功能全景）、第 4 章（目标用户）、第 6 章（技术约束）、`tech-stack.md`

### 内容要点

| 内容 | 说明 |
|------|------|
| 产品一句话定义 | PRD 第 2 章 `{one_liner}` |
| 技术栈清单 | 后端语言/框架、前端框架、数据库、AI 服务 |
| 项目目录结构 | 关键路径和职责说明 |
| 编码约定 | 格式化工具、命名规范、测试框架 |
| 开发环境搭建步骤 | 简明的本地开发启动命令 |
| 关键依赖列表 | 框架、库、工具链 |

### 格式要求

```markdown
# 项目上下文：{产品名称}

## 产品简介
{one_liner}

## 技术栈
- 后端：{tech_stack}
- 前端：{frontend_stack}
- AI：{llm_service}
- 数据库：{database}

## 目录结构
```
backend/
  main.py          # FastAPI 入口
  routes/          # API 路由
  services/        # 业务逻辑
frontend/
  src/
    pages/         # 页面组件
    components/    # 通用组件
```

## 编码约定
- Python 使用 Black 格式化 + ruff 检查
- TypeScript 使用 ESLint + Prettier
- 命名：变量/函数用 camelCase，类用 PascalCase，文件用 kebab-case

## 开发环境
```bash
# 后端
cd backend && pip install -r requirements.txt && uvicorn main:app --reload

# 前端
cd frontend && npm install && npm run dev
```
```

## 2. 功能开发指导

**用途**：指导后端 AI 编程工具实现某个具体功能的后端逻辑。

**数据来源**：PRD 第 3 章（MVP 功能全景 — 功能名称和优先级）、第 5 章（交互流程与界面设计 — 交互步骤）、第 9 章（验收标准 — DoD）

### 内容要点

| 内容 | 说明 |
|------|------|
| 功能名称和优先级 | 从 PRD 第 3 章 |
| 功能描述（一句话） | 该功能做什么 |
| 关联页面 | 对应 PRD 第 5 章的哪个页面 |
| 交互流程步骤 | 从 PRD 第 5 章逐条展开，按用户操作顺序 |
| 需满足的验收条件 | 从 PRD 第 9 章该功能的 DoD |
| 边界场景处理 | 数据为空、错误状态、并发等 |
| 输入/输出示例 | 典型的数据样例 |

### 格式要求

每个功能一个独立文件，按功能名命名：`feature-{序号}-{功能英文名}.md`。

```markdown
# 功能：{功能名称}

## 优先级
P{0/1/2}

## 功能描述
一句话说明

## 交互流程
1. 用户进入页面 → 看到 {界面元素}
2. 用户 {操作} → 系统 {响应}
3. ...

## 验收标准
- [ ] 条件 1：{可验证的条件}
- [ ] 条件 2：{可验证的条件}

## 边界场景
- 场景 A：{描述} → 处理方式：{方案}
- 场景 B：{描述} → 处理方式：{方案}

## 数据示例
```json
{ ... }
```
```

## 3. 接口实现指导

**用途**：指导后端 AI 编程工具按接口文档实现具体的 API 端点。

**数据来源**：接口文档第 3 章（接口清单 — 每个接口的定义）、第 4 章（数据模型 — 结构体定义）、第 2 章（约定 — 响应格式和错误码）

### 内容要点

| 内容 | 说明 |
|------|------|
| 接口名称和路径 | 从接口文档 |
| 请求/响应格式 | 完整 JSON 结构 |
| 业务逻辑说明 | 对应 PRD 中的哪个交互步骤、处理规则是什么 |
| 数据校验规则 | 必填、枚举、长度限制 |
| 错误码处理 | 该接口可能的错误码及触发条件 |
| LLM 调用集成（如有） | 如何调用 AI 服务、prompt 模板引用 |
| 测试用例建议 | 正常路径 + 异常路径 |

### 格式要求

每个接口一个独立文件，按路径命名：`api-{method}-{资源路径}.md`。

```markdown
# 接口实现：{接口名称}

## 定义
{METHOD} /api/{path}

## 请求
{请求体或参数结构}

## 响应
```json
{
  "code": 0,
  "data": { ... },
  "message": "ok"
}
```

## 业务逻辑
1. 校验参数 → 不满足则返回 {错误码}
2. 查询/更新数据 → 记录到 SessionData
3. 如需 LLM 调用 → 构建 prompt → 触发 SSE 流式输出
4. 返回结果

## 校验规则
| 字段 | 规则 | 错误码 |
|------|------|--------|
| {字段} | {规则} | {错误码} |

## 错误码
| 码 | 条件 |
|----|------|
| {code} | {触发条件} |

## 测试用例
- TC1：正常路径 → 期望 {结果}
- TC2：{字段} 缺失 → 期望 40001
- TC3：{字段} 非法值 → 期望 40002
```

## 4. 前端开发指导

**用途**：指导前端 AI 编程工具实现前端页面和组件。

**数据来源**：PRD 第 5 章（交互流程与界面设计 — 页面布局和交互步骤）、接口文档第 3 章（接口清单 — 接口列表）、第 5 章（业务流程对照 — 接口调用顺序）

### 内容要点

| 内容 | 说明 |
|------|------|
| 页面路径和名称 | 从 PRD 第 5 章 |
| 页面组件分解 | 页面拆分为哪些子组件 |
| 数据流 | 页面需要调用哪些接口、数据如何流向组件 |
| 交互逻辑 | 按钮点击 → 调用接口 → 更新状态 → 渲染结果 |
| 加载/空/错误状态 | 三种状态的 UI 处理 |
| 路由配置 | 前端路由路径和参数（含 session_id） |

### 格式要求

每个页面一个独立文件，按页面名命名：`page-{页面英文名}.md`。

```markdown
# 前端页面：{页面名称}

## 路由
`/{path}` 或 `/{path}/:session_id`

## 组件树
```
{页面名称}
├── 组件 A
│   └── 子组件 A1
├── 组件 B
└── 组件 C
```

## 数据流
- 页面加载 → `GET /api/xxx` → 渲染组件 A/B
- 用户操作 → `POST /api/xxx` → 更新状态 → 组件 C 重新渲染

## 接口依赖
| 操作 | 接口 | 说明 |
|------|------|------|
| 页面初始化 | `GET /api/xxx` | 加载数据 |
| 提交操作 | `POST /api/xxx` | 创建资源 |

## 状态处理
| 状态 | UI |
|------|----|
| 加载中 | `{加载组件描述}` |
| 数据为空 | `{空状态描述}` |
| 错误 | `{错误提示描述}` |
| 正常 | `{正常渲染描述}` |

## 路由参数
| 参数 | 来源 | 说明 |
|------|------|------|
| `{param}` | URL/State | `{说明}` |
```

## 5. 代码审查指导

**用途**：给代码审查者或 AI 编程工具做代码质量检查时使用。不是生成代码用的，是检查代码用的。

**数据来源**：PRD 第 7 章（非功能性需求 — 性能/安全标准）、第 9 章（验收标准 — 各功能 DoD）、技术约束 + 编码约定

### 内容要点

| 内容 | 说明 |
|------|------|
| 功能完整性检查 | 是否满足 PRD 第 9 章的 DoD |
| API 一致性检查 | 接口实现是否与接口文档一致 |
| 边界场景覆盖 | 数据为空、错误、并发等场景是否处理 |
| 非功能需求检查 | 性能、安全、兼容性要求是否满足 |
| 代码规范检查 | 命名、格式化、目录结构是否符合约定 |
| Review Checklist | 可逐项勾选的审查列表 |

### 格式要求

一份综合文档，按模块分节。文件名 `code-review-guide.md`。

```markdown
# 代码审查指导：{产品名称}

## 通用审查项
- [ ] 目录结构是否与约定一致
- [ ] 命名规范是否符合（camelCase / PascalCase / kebab-case）
- [ ] 错误处理是否覆盖了所有 {错误码}
- [ ] 日志是否完整（关键路径有日志、异常有堆栈）

## 接口实现审查
- [ ] 请求参数校验是否完整
- [ ] 返回格式是否符合 `{ "code": 0, "data": ... }`
- [ ] 错误码是否与接口文档约定一致

## 功能审查
按功能逐一检查：

### {功能名称}
- [ ] 功能路径：用户 {操作} → 系统 {响应} 是否实现
- [ ] DoD 全部满足：
  - [ ] {条件 1}
  - [ ] {条件 2}

### {功能名称}
（同上结构）

## 非功能审查
| 维度 | 检查项 | 预期 |
|------|--------|------|
| 性能 | {项} | {预期值} |
| 安全 | {项} | {预期值} |
| 兼容性 | {项} | {预期值} |

## 边界场景审查
- [ ] 数据为空时的 UI 和 API 行为
- [ ] 网络错误 / LLM 超时的降级处理
- [ ] 并发请求（如快速连点提交按钮）
```

  api-get-{资源}-id.md
  api-post-{资源}-id-confirm.md
pages/
  page-{页面1}.md                      # 前端开发指导（K 份）
  page-{页面1}-chat.md
  page-{页面2}.md
code-review-guide.md                   # 代码审查指导（1 份）
```

> **文件命名规则**：英文小写 + 连字符，与 PRD 和接口文档中的路径/名称对齐。
