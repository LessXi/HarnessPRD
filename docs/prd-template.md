# PRD（产品需求文档）模板

> 版本 1.0 · 2026-06-15
>
> 目标读者：开发团队
> 文档目标：帮助开发团队理解产品需求，指导后续开发工作

---

## 第 1 章：文档信息

| 字段 | 内容 |
|------|------|
| 文档名称 | `{产品名称}` PRD |
| 版本号 | v1.0 |
| 更新日期 | `{生成日期}` |
| 文档状态 | 初稿 / 已审核 / 定稿 |
| 文档概述 | 本文档旨在描述 `{产品名称}` 的产品需求，包括产品背景、目标用户、功能定义、交互流程、技术约束等内容，作为后续开发工作的依据。 |

**写作风格**：概要——仅元数据，无需展开。

**数据来源**：系统自动生成（产品名、时间戳）+ `doc_depth` 控制后续章节篇幅。

---

## 第 2 章：产品概述

**内容要点**：
- **一句话定义**：`{one_liner}`
- **核心痛点**：`{problem_statement}`，扩展为 2-3 段场景描述，说明痛点出现的频率、影响范围和当前的替代方案
- **成功指标**：如何衡量产品是否解决了问题（由 AI 从对话中推断，如"用户完成一份 PRD 的时间从 4 小时缩短到 1 小时"）

**写作风格**：简洁——300 字以内，让读者快速理解"这是做什么的"。

**数据来源**：`form_data` + 对话历史（阶段 1 的痛点讨论）。

---

## 第 3 章：MVP 功能全景

**内容要点**：
- MVP 功能总览表，按优先级排序：

| 优先级 | 功能名称 | 一句话说明 | 所属页面 |
|--------|---------|-----------|---------|
| P0 | `{功能1}` | `{说明}` | `{页面}` |
| P0 | `{功能2}` | `{说明}` | `{页面}` |
| P1 | `{功能3}` | `{说明}` | `{页面}` |
| ... | ... | ... | ... |

- 优先级定义：P0（必须有）/ P1（重要）/ P2（锦上添花）
- 功能之间的依赖关系说明（如有）

**写作风格**：概要——表格为主，快速呈现功能全景。每功能一句话，不展开细节。

**数据来源**：`mvp_features` + `feature_priority` + `doc_depth`。

---

## 第 4 章：目标用户

**内容要点**：
- **核心用户画像**：`{target_users}`，扩展为身份 / 痛点 / 使用频次 / 技术背景
- **典型使用场景**：1-2 个用户故事（格式：As a… I want… So that…）
- **使用场景还原**：用户在实际工作中遇到的具体故事（来自对话阶段 1 的讨论）
- **非目标用户**：明确什么人不适合这个产品

**写作风格**：简洁——每段 2-3 句话。使用故事化描述增强代入感。

**数据来源**：`target_users` + 对话阶段 1（用户讲述的真实场景）。

---

## 第 5 章：交互流程与界面设计

> 本章是 PRD 的核心章节，以页面/功能为维度组织内容。每个页面自包含：长什么样 + 怎么交互 + 边界情况。

**内容要点**（每个页面一个子章节）：

### 5.1 `{页面路径}` — `{页面名称}`

- **页面布局描述**：
  - 页面包含哪些模块/区域，各区域的位置关系
  - 视觉风格参考（`{visual_style}`）
  
- **交互流程**（按操作顺序）：
  1. 用户进入页面 → 看到什么
  2. 用户执行操作 A → 系统响应是什么
  3. 用户执行操作 B → 系统响应是什么
  4. …

- **边界场景**：
  - 数据为空时显示什么
  - 网络错误时怎么处理
  - 用户输入无效值时如何反馈
  - 操作耗时较长时的加载状态

### 5.2 `{页面路径}` — `{页面名称}`

（同上结构）

**写作风格**：详细——每页面半页到一页。交互流程用有序列表分步描述，边界场景用"场景 → 处理方式"列表。开发可直接据此估算工作量。

**数据来源**：对话阶段 2（功能深挖）+ `page_count` + `visual_style` + 功能清单推导。

## 第 6 章：技术约束

**内容要点**：

| 维度 | 决策 | 说明 |
|------|------|------|
| 目标平台 | `{platform_type}` | — |
| 认证方式 | `{needs_auth}` | 如"需要，邮箱注册 + 第三方登录" |
| 数据存储 | `{needs_database}` | 如"需要，存储用户配置和生成的文档" |
| 页面数量 | `{page_count}` | 对应第 5 章的页面清单 |
| 技术栈偏好 | `{tech_stack_preference}` | 用户的倾向和限制 |
| 时间预期 | `{timeline_expectation}` | 对 MVP 范围的影响说明 |
| 额外上下文 | `{additional_context}` | 如"单人开发，预算有限"等 |

**写作风格**：概要——表格或列表，条理清晰即可。

**数据来源**：表单字段 + 对话阶段 4 的讨论结果。若对话中有修改，优先采用对话确认版。

---

## 第 7 章：非功能性需求

**内容要点**：
- **性能预期**：页面加载时间目标、并发用户估算、API 响应时间要求
- **可用性**：关键功能可用率目标（如 99.9%）
- **兼容性**：支持的浏览器 / 设备 / 操作系统
- **AI 生成质量**：`{ai_temperature}` 对应的输出风格（保守 / 平衡 / 创意）
- **安全性**：数据隐私要求、是否需要 HTTPS、敏感信息处理

**写作风格**：简洁——每点 1-2 句话。信息不足时标注 `[需补充]`，不编造内容。

**数据来源**：`ai_temperature` + `additional_context` + 对话推断。大部分信息用户未直接填写，AI 需根据项目类型合理推断或标记待补充。

---

## 第 8 章：竞品与差异化

**内容要点**：
- **主要竞品**：`{competitors}`（如有），按"产品名称 → 核心特点 → 强项/弱项"列出
- **核心竞争力**：对话阶段 3 提炼的差异化定位
- **差异化关键词**：2-3 个关键词（如"AI 驱动 / 对话式 / 一键生成"）
- **市场定位**：一句话说明产品在竞品格局中的位置

**写作风格**：简洁——竞品用表格，定位用 2-3 句话总结。

**数据来源**：`competitors` + 对话阶段 3。若用户未填竞品，AI 从对话推断是否有同类产品。

---

## 第 9 章：验收标准

**内容要点**：
- 总体标准：MVP 全部功能完成 → 可进入下一阶段
- 每个功能的 DoD（Definition of Done）：

| 功能 | DoD |
|------|-----|
| `{功能1}` | ① `{条件1}` ② `{条件2}` ③ … |
| `{功能2}` | ① `{条件1}` ② `{条件2}` ③ … |
| … | … |

- 文档质量要求：PRD 内容结构完整、可读性高、开发团队可直接依据

**写作风格**：详细——按功能逐条列出，条件应清晰可验证（能回答"是/否"）。

**数据来源**：AI 根据对话和功能清单推断。条件应具体（如"用户可成功提交表单并跳转到对话页"），而非模糊（如"体验好"）。

---

## 第 10 章：开放问题

**内容要点**：
- 对话阶段 5 标注的矛盾点（标注 `[⚠️ 待确认]`）
- 信息不足以决策的事项（标注 `[需补充]`）
- 需要业务方或产品负责人进一步确认的决策

**示例**：
- `[⚠️ 待确认]` 功能 A 的交互方案与用户最初描述的痛点不完全匹配，需确认是否接受
- `[需补充]` 未明确预期并发用户数，建议上线前补充压测目标

**写作风格**：概要——列表即可，每项说明原因和需要的决策。

**数据来源**：对话阶段 5 的矛盾检测结果 + AI 识别到的信息缺口。
